# HighPerformanceBlacksmith 架构升级说明

## 1. 概述

HighPerformanceBlacksmith (HPB) 是 BlacksmithFramework (BF) 的高性能演进版本。本次升级的核心目标是**将判定管线中的"数据"与"行为"解耦**，从"数据自带执行逻辑"的 Resolution 模式，转变为"纯数据 + 可注册分析器"的 AnalyzableData 模式。

### 当前进度

| 组件 | 状态 |
|---|---|
| 核心引擎（AnalyzableData、AnalyzerRegistry、TurnContext、9 阶段管线） | ✅ 已完成 |
| StandardAnalyzers（DSL 5 个 + Defense 6 个 = 11 个默认分析器） | ✅ 已完成 |
| DefenseEntity 防御体系（7 种防御类型 + 6 个分析器） | ✅ 已完成 |
| Common 职业 DSL 迁移 | ✅ 已完成 |
| Warlock 职业 DSL 迁移 | ✅ 已完成 |
| Driver 职业 DSL 迁移 | ✅ 已完成 |
| Cannon 职业 DSL 迁移 | ✅ 已完成 |
| BloodSigil 职业 DSL 迁移 | ✅ 已完成 |
| Alchemy 职业 DSL 迁移（装备技能） | ✅ 已完成 |
| Lancer 职业 | ❌ 未迁移（代码已注释） |
| ModExamples（7 个示例 Mod） | ⚠️ 部分迁移 |

### 核心理念变化

```
旧:  Skill → Resolution（数据 + Execute 委托）→ 直接执行
新:  Skill → AnalyzableData（纯数据 + AnalyzerKey）→ AnalyzerRegistry 查找 → 执行
```

- **旧架构**：每个 Resolution 对象内部携带 `Execute` 委托，数据对象知道如何执行自己。攻击的穿甲计算、防御的减伤逻辑全部硬编码在闭包中。
- **新架构**：AnalyzableData 是纯数据载体，仅携带 `AnalyzerKey` 字符串。所有执行逻辑归属于 `AnalyzerRegistry` 中注册的静态分析器方法，由 `StandardAnalyzers` 提供默认实现。

## 2. 核心架构变更：Resolution → AnalyzableData

### 2.1 概念对比

| 维度 | 旧 (BF) | 新 (HPB) |
|---|---|---|
| 数据接口 | `IResolution` | `IAnalyzableData` |
| 数据类命名 | `XxxResolution` | `XxxAnalyzableData` |
| 执行方式 | 数据自带 `Execute` 委托 | `AnalyzerKey` → `AnalyzerRegistry` |
| 数据类型感知 | Resolution 自己知道如何处理 | 由 Analyzer 根据类型处理 |
| 目录 | `Infra/Models/Components/Resolutions/` | `Infra/Models/Components/AnalyzableDatas/` |

### 2.2 文件映射

| 旧文件 (BF) | 新文件 (HPB) | 说明 |
|---|---|---|
| `Resolutions/AttackResolution.cs` | `AnalyzableDatas/AttackAnalyzableData.cs` | 攻击数据 |
| `Resolutions/DefenseResolution.cs` | `AnalyzableDatas/DefenseAnalyzableData.cs` | 防御数据 |
| `Resolutions/ResourceResolution.cs` | `AnalyzableDatas/ResourceAnalyzableData.cs` | 资源数据 |
| `Resolutions/EffectResolution.cs` | `AnalyzableDatas/EffectAnalyzableData.cs` | 效果数据 |
| `Particular/EffectEntity.cs` | `AnalyzableDatas/EffectEntity.cs` | 效果实体（新：实现 IAnalyzableData） |

### 2.3 数据类对比（以 Attack 为例）

**旧 — AttackResolution（数据+行为一体）:**
```csharp
public class AttackResolution : IResolution
{
    public Action<Community> Execute { get; set; } = (a) => { };
    // Execute 内部包含完整的攻击流程：遍历防御 → 穿甲计算 → 扣血
    // 在 WriteAttack 时通过闭包构建
}
```

**新 — AttackAnalyzableData（纯数据）:**
```csharp
public class AttackAnalyzableData : IAnalyzableData
{
    public required string AnalyzerKey { get; init; }      // 指向 AnalyzerRegistry
    public required ClapRoundClock Clock { get; init; }
    public required AttackType.CEValue Type { get; init; }
    public required int Power { get; set; }
    public required float APFactor { get; init; }
    public int TotalDamage { get; set; } = 0;
    public Dictionary<AttackStage.CEValue, List<string>> StageKeys { get; init; } = new();
}
```

### 2.4 TurnContext 泛型参数变更

```csharp
// 旧
public class TurnContext : ClapTurnContext<IResolution, Community>

// 新
public class TurnContext : ClapTurnContext<IAnalyzableData, Community>, IComponent<Body>
```

新 TurnContext 同时实现了 `IComponent<Body>`，与 Body 有了直接引用关系，便于访问 Body 上的其他组件。

## 3. 防御系统重构：DefenseBase → DefenseEntity

### 3.1 架构对比

**旧 — DefenseBase（行为嵌入子类）:**
```csharp
public abstract class DefenseBase : IDefenseWork
{
    public abstract (int, int) Work(Body source, Body owner, int attack, AttackType.CEValue type);
    // 每个子类必须实现 Work()，返回 (remainingAttack, damageDealt)
}
```

每个防御子类（CommonArmor、CommonReduction 等）必须实现自己的 `Work()` 方法：
```csharp
// 旧 CommonArmor — 行为嵌入
public class CommonArmor : DefenseBase
{
    public override (int, int) Work(Body source, Body owner, int attack, AttackType.CEValue type)
    {
        var res = (Math.Max(0, attack - Power), (int)MathF.Min(attack, Power));
        Power = Math.Max(0, Power - attack);
        return res;
    }
}
```

**新 — DefenseEntity（纯数据）:**
```csharp
public abstract class DefenseEntity : IAnalyzableData
{
    public abstract string AnalyzerKey { get; init; }    // 指向 AnalyzerRegistry.Defense
    public abstract DefenseType.CEValue Type { get; init; }
    public abstract int Power { get; set; }
    public abstract ClapRoundClock Clock { get; init; }
    public virtual bool CanMerge { get; init; } = false;
    public virtual void Merge(DefenseEntity addition) { }
    public virtual void Update() { /* Clock 推进 & 死亡判定 */ }
}
```

防御子类变为纯粹的数据声明：
```csharp
// 新 CommonArmor — 仅数据
public class CommonArmor : DefenseEntity
{
    public override string AnalyzerKey { get; init; } = nameof(StandardAnalyzers.DefaultArmor);
    public override DefenseType.CEValue Type { get; init; } = DefenseType.Instance.CommonArmor();
    public override float Power { get; set; } = 0;
    public override ClapRoundClock Clock { get; init; } = new(isInfinite: true);
}
```

### 3.2 防御计算逻辑集中化

所有防御的计算逻辑移至 `StandardAnalyzers`：

```csharp
// StandardAnalyzers.Defense.cs
public partial class StandardAnalyzers : MainProfession
{
    [IsAnalyzer(AnalyzerType.Defense)]
    public static void DefaultArmor(Community player, Community enemy,
        DefenseEntity defense, AttackAnalyzableData attackData)
    {
        var damage = Math.Min(attackData.Power, defense.Power);
        defense.Power = Math.Max(0, defense.Power - attackData.Power);
        attackData.Power -= damage;
        attackData.TotalDamage += damage;
    }
    // DefaultReduction, ThornReduction, MagicalImmunity, PhysicalImmunity, PercentageReduction ...
}
```

### 3.3 关键收益

- **新增防御类型极简**：只需声明数据类（继承 `DefenseEntity`）+ 注册 Analyzer，无需重复实现计算逻辑
- **行为可动态替换**：更换 `AnalyzerKey` 即可改变防御行为，无需修改防御类本身
- **计算逻辑集中审计**：所有防御计算在 `StandardAnalyzers` 中一目了然
- **性能优化空间**：纯数据更利于缓存和批量处理

### 3.4 删除的文件

| 文件 | 原因 |
|---|---|
| `Infra/Models/Core/DefenseBase.cs` | 被 `AnalyzedObjects/DefenseEntity.cs` 取代 |
| `Infra/Models/Core/IDefenseWork.cs` | 行为接口不再需要，逻辑移至 Analyzer |

## 4. Analyzer 注册体系（全新基础设施）

### 4.1 架构概览

```
┌─────────────────────────────────────────────────┐
│                  AnalyzerRegistry                │
├─────────────────────────────────────────────────┤
│  .DSL          AnalyzerRegistry<Action<C,C,     │
│                IAnalyzableData>>                 │
│  .Defense      AnalyzerRegistry<Action<C,C,     │
│                DefenseEntity, AttackAnalyzable>> │
│  .JudgeCallback AnalyzerRegistry<Action<C,C>>   │
│  .Universal    UniversalRegistry                │
└─────────────────────────────────────────────────┘
          ▲                ▲
          │ 注册            │ 查找
          │                │
  ┌───────┴──────┐  ┌──────┴──────┐
  │ [IsAnalyzer] │  │ AnalyzerKey │
  │ 静态方法      │  │ 字符串       │
  └──────────────┘  └─────────────┘
```

### 4.2 新增文件

| 文件 | 说明 |
|---|---|
| `Infra/DSL/AnalyzerRegistry.cs` | 全局静态注册表，4 种类型的分析器注册中心 |
| `Infra/DSL/StandardAnalyzers.DSL.cs` | 默认 DSL 分析器：DefaultAttack, DefaultResource, DefaultDefense |
| `Infra/DSL/StantardAnalyzers.Defense.cs` | 默认防御分析器：DefaultArmor, DefaultReduction, ThornReduction 等 |
| `Infra/Attributes/Profession/IsAnalyzer.cs` | `[IsAnalyzer]` 属性 + `AnalyzerType` 枚举 |

### 4.3 AnalyzerType 枚举

```csharp
public enum AnalyzerType
{
    DSL,           // 通用：处理 Attack/Defense/Resource/Effect 的写入
    Defense,       // 防御专属：签名含 DefenseEntity + AttackAnalyzableData
    JudgeCallback, // 判定回调
    Universal      // 泛型注册
}
```

### 4.4 [IsAnalyzer] 属性

```csharp
[AttributeUsage(AttributeTargets.Method, AllowMultiple = false, Inherited = false)]
public class IsAnalyzer : Attribute
{
    public readonly AnalyzerType Type;
    public IsAnalyzer(AnalyzerType type = AnalyzerType.DSL) { Type = type; }
}
```

标记了 `[IsAnalyzer]` 的静态方法在 Mod 加载时被自动扫描并注册到对应的 `AnalyzerRegistry` 中。

### 4.5 默认分析器一览

| Analyzer | 类型 | 功能 |
|---|---|---|
| `DefaultAttack` | DSL | 攻击流程：遍历防御 → AP 因子 → 扣血 |
| `DefaultResource` | DSL | 资源获取 |
| `DefaultDefense` | DSL | 防御添加 |
| `DefaultArmor` | Defense | 护甲吸收伤害（护甲值同步减少） |
| `DefaultReduction` | Defense | 减伤（值不减少，纯吸收） |
| `ThornReduction` | Defense | 荆棘减伤（反弹一半伤害为魔法攻击） |
| `MagicalImmunity` | Defense | 魔法免疫 |
| `PhysicalImmunity` | Defense | 物理免疫 |
| `PercentageReduction` | Defense | 百分比减伤 |

## 5. 判定管线变更

### 5.1 管线精简：11 阶段 → 9 阶段

删除了 OnEffectSwaping 和 OnAttackSwaping 两个阶段，简化判定流程。

### 5.2 完整对比

| 序号 | 旧 Stage (BF) | 值 | 新 Stage (HPB) | 值 | 变更 |
|---|---|---|---|---|---|
| 1 | OnBegin | 0 | OnBegin | 0 | 不变 |
| 2 | OnEffectSwaping | 8 | — | — | **移除** |
| 3 | OnApplyingEffect | 16 | OnApplyingEffect | 8 | 值重排 |
| 4 | OnEffectTaking\_AfterResolutionWritten | 32 | OnEffectTaking\_AfterAnalyzableDataWritten | 16 | 重命名 + 值重排 |
| 5 | OnAttackCanceling | 64 | OnAttackCanceling | 32 | 值重排 |
| 6 | OnAttackSwaping | 128 | — | — | **移除** |
| 7 | OnEffectTaking\_AfterTransport | 256 | OnEffectTaking\_AfterTransport | 64 | 值重排 |
| 8 | OnApplyingOthers | 512 | OnApplyingOthers | 128 | 值重排 |
| 9 | OnUpdating | 1024 | OnUpdating | 256 | 值重排 |
| 10 | OnEffectTaking\_AfterResult | 2048 | OnEffectTaking\_AfterResult | 1024 | 值重排 |
| 11 | OnEnd | 4096 | OnEnd | 2048 | 值重排 |

### 5.3 关键变更说明

- **移除 OnEffectSwaping**：效果交换逻辑不再作为独立阶段，合并到 Effect 的 Analyzer 处理中
- **移除 OnAttackSwaping**：攻击交换逻辑不再作为独立阶段
- **重命名 OnEffectTaking_AfterResolutionWritten → OnEffectTaking_AfterAnalyzableDataWritten**：反映底层数据模型的变化

### 5.4 判定执行方式变化

**旧 — Effect 自带 Execute:**
```csharp
// JudgeRuleManager.TakeEffects (BF)
player.Focus.Get<Effect>().Execute(type, player.Focus);
// Effect.Execute 内部调用 effectEntity.Execute(body)
```

**新 — AnalyzerRegistry 调度:**
```csharp
// JudgeRuleManager.TakeEffects (HPB)
foreach (var entity in player.Focus.Get<Effect>().Where(type))
{
    if (entity.Clock.IsRinging)
    {
        AnalyzerRegistry.DSL.Get(entity.AnalyzerKey)(player, enemy, entity);
    }
}
```

## 9. 设计原则总结

1. **数据与行为分离**：AnalyzableData 只描述"发生了什么"，Analyzer 决定"如何处理"
2. **开闭原则**：新增游戏机制（攻击类型、防御类型、效果类型）通过注册新 Analyzer 完成，无需修改现有代码
3. **可替换性**：通过更换 `AnalyzerKey` 即可在运行时切换行为，例如将防御的减伤算法从固定值改为百分比
4. **集中管理**：所有默认行为在 `StandardAnalyzers` 中统一可见，便于审计、测试和优化
5. **减少委托分配**：新架构减少了每个 Resolution 对象上的闭包和委托分配，改用静态方法引用，降低 GC 压力

## 10. 性能数据

### 10.1 测试环境

- **配置**：VS 发布模式（Debug build）
- **测试方式**：Client对战场景（Adversarial AI 对战多回合）
- **对比基准**：BlacksmithFramework (BF) 同等场景

### 10.2 测试结果

| 指标 | BF (Debug) | HPB (Debug) | 提升倍数 |
|---|---|---|---|
| 每回合判定耗时 | 基准 | ~1/10 | ~9-10x |
| 委托分配次数 | 每 Resolution 至少 1 次 | 0（静态方法引用） | 消除 |
| GC 压力 | 高频闭包分配 | 大幅降低 | 显著改善 |

> **注意**：复杂场景（多 Mod、多召唤物）的性能数据待补充。

### 10.3 性能提升来源

1. **消除委托分配**：旧架构每个 `AttackResolution`、`DefenseResolution`、`EffectResolution` 携带至少一个 `Action<...>` 闭包委托；新架构使用静态方法引用，零分配
2. **int 替代 float**：攻击和防御 Power 从 `float` 改为 `int`，减少浮点运算和类型转换
3. **管线精简**：11 阶段 → 9 阶段，减少两次全量遍历
4. **数据局部性**：纯数据对象更小、更利于 CPU 缓存
5. **静态分发**：`AnalyzerRegistry` 的字典查找虽然是间接调用，但目标方法是固定的静态方法，JIT 可以更好地内联优化
