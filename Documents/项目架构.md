# 项目架构

面向维护者、调试者和 Mod 开发者。本文档描述 HighPerformanceBlacksmith 的完整架构。

---

## 1. 项目概览

多个 .NET 8 项目组成的方案。

```
BlacksmithCore  (自包含，无外部项目依赖)
  ↑
BlacksmithClient  BlacksmithServer  AIPVPPlatform

ModExamples  (→ BlacksmithCore，示例 Mod)
ClapSourceGenerators  (Roslyn 编译时分析器，独立项目)
```

BlacksmithClient 启动 ASP.NET Core 本地站点，用 `wwwroot` 托管静态页面，通过 `/api/*` 端点操作同一进程里的 `GameInstance`。

## 2. 核心基础设施

BlacksmithCore 自包含所有基础设施。

### 可扩展枚举：`BlacksmithEnum<T, TMemberAttribute>`

C# 原生 `enum` 编译后不可追加。`BlacksmithEnum` 通过"运行时创建成员 + 全局递增唯一 ID"解决。

成员通过 `[IsBlacksmithEnumMember(priority)]` 特性的 `public` 无参实例方法自动注册。`Create(name, priority)` 直接覆盖同名成员——这是 Mod 修改枚举优先级的基础。启动完成后 `CloseFactory()` 关闭工厂。

核心可扩展枚举类型：`ResourceType`、`AttackType`、`DefenseType`、`EffectType`、`EffectTargetType`、`JudgeStage`、`AttackStage`。

所有可扩展枚举均有不可被覆盖的成员Default()，作为无效值使用。

### 技能包：`SkillPackageBase`

编译时由 Roslyn 增量源生成器扫描 `private` 实例方法（或 `static` 方法）并配对：

- `XxxCheck(ISkillContext)` → `bool`：可用性检查
- `Xxx(ISkillContext)` → `IDSLSourceFile`：技能实现

诊断级别为 **Error**：未配对的 Check 方法或错误的返回类型会阻止编译。

### 多包技能管理：`Skill`

管理多个技能包，提供统一入口：`AddPackage` / `AddSkill` / `RemoveSkill` / `TryDeclare` / `Declare`。`PackageContainer` 携带 `ClapSharedFlag` 实现包的启用/禁用。

### 实体组件：`Body`

按类型键存储组件的容器。`Get<T>()` 取出组件。`Update()` 遍历 `IUpdatePerRound` 组件。

`Body` 注入 7 个组件：`Skill`、`Health`、`Defense`、`Resource`、`Effect`、`TurnContext`、`Community`。

### 回合上下文：`TurnContext`

泛型参数 `IAnalyzableData`，按类型缓存 `List<TAnalyzableData>`，同时实现 `IComponent<Body>` 与 `IUpdatePerRound` 以直接访问 Body 并管理预处理钩子的生命周期。

核心方法：

- `WriteAnalyzableData(analyzableData)`：先运行预处理钩子，再写入列表
- `Get<TAnalyzableData>()`

注册四种数据：`AttackAnalyzableData`、`DefenseAnalyzableData`、`ResourceAnalyzableData`、`EffectAnalyzableData`。每个数据携带一个 `ClapRoundClock`，执行时只处理 `Clock.IsRinging` 的条目，否则调用 `Clock.RoundPass()` 推进。

### 工具

- **`DllLoader`**：扫描目录下所有 `*.dll`，提供 `LoadByType<T>()` 和 `LoadStaticByAttribute()`
- **`BlacksmithEnumRegistry`**：枚举注册表
- **`ClapSourceGenerators`**：Roslyn 增量源生成器，含技能配对、分析器自动注册、Modifier 绑定等
- **`ClapRoundClock`**：统一的多回合生命周期时钟（详见[判定流程](./判定流程.md#9-claproundclock-时钟机制)）
- **`ClapSharedFlag`**：引用计数式启用/禁用开关

## 3. 启动流程

入口：`Program.cs`。

```
ModLoader.Initialize(BaseDirectory)
  1. 读取 .blacksmith/mod.json 解析 Mod 目录
  2. DllLoader 扫描目录下 *.dll
  3. 加载 BlacksmithEnum 子类和 IsBlacksmithEnumModifier 静态类 → 注册枚举 → CloseFactory()
  4. 扫描 SkillPackageBase 子类 → 注册主职业名 → 注册修改器
  5. 扫描 [IsAnalyzer] 静态方法 → 注册到 AnalyzerRegistry
  6. 扫描 ISkillMetadata 实现类 → 收集元数据
  7. AddModOnInit 将修改器技能合并到目标包 + 调用 modifier.Bind(package)
→ LocalHost.Start(strategies)
  配置 ASP.NET Core，绑定 http://localhost:5000，映射 API，打开浏览器
```

## 4. 运行时核心对象

### GameInstance

一局游戏的根容器，实现 `ISkillContext` 和 `ISudoOperations`。核心属性：

- `Community Player` / `Community Enemy`：双方阵营
- `Judger Judger`：回合裁判
- `GameHistory History`：历史记录
- `GameMetadata Metadata`：全局元数据

### Community

`Body Focus` + `List<Body> SummonList`。支持召唤/分身等多角色机制：

- `AddTransform(action, delayRounds, remainingRounds, isInfinite)`：注册定时行为
- `AddCallbackKilled(body, callback)`：注册召唤物死亡回调
- `Update()`：清理已死亡召唤物 → 推进 Transform 时钟 → 更新 Focus 和各 Summon Body

### Body → BodyView

`Body` 通过 `Get<T>()` 访问 7 个组件。`GetView()` 返回 `BodyView` 供前端使用。

### 组件一览

| 组件 | 职责 |
|---|---|
| `Health` | HP / MHP，`GainHP` / `LoseHP` / `GainMHP` / `LoseMHP` |
| `Resource` | 资源系统，`Check` / `Use` / `Gain` / `Query` |
| `Defense` | `List<DefenseEntity>`，`Add`（同类型且 `CanMerge` 则合并），`Update` 清理过期防御 |
| `Effect` | `List<EffectEntity>`（每个持有 `ClapRoundClock`），`Update` 清理 `Clock.IsDead` 的过期效果 |
| `TurnContext` | 四种 AnalysableData 缓存，通过 `ClapRoundClock` 延迟处理，预处理钩子（带独立 Clock 生命周期），`Update()` 清理过期预处理 |
| `Skill` | 多包技能管理 |
| `Community` | Focus + SummonList |

## 5. 技能系统

### SkillPackageBase

编译时源生成器自动配对 `XxxCheck` / `Xxx`。技能名转为小写。

### MainProfession 与 ProfessionModifier

- **`MainProfession`**：抽象主职业基类，可包含 `[IsAnalyzer]` 静态方法
- **`ProfessionModifier`**：抽象修改器基类，标注 `[IsProfessionModifier("目标名")]`

**ModifierBindingGenerator**：Roslyn 增量源生成器，使 Modifier 能够直接读写目标职业的私有状态变量，零反射开销。

### SkillMetadata 技能元数据

| 属性 | 说明 |
|---|---|
| `[HasAttack(float power)]` | 标记为攻击技能，携带攻击力信息 |
| `[HasDefense]` | 标记为防御技能 |
| `[HasResource]` | 标记为资源技能 |
| `[HasRecovery]` | 标记为恢复技能 |
| `[HasBuff]` | 标记为 Buff 技能 |
| `[IsProfessionSkill]` | 标记为转职入口技能 |
| `[IsEquipmentSkill]` | 标记为装备技能 |
| `[Labels(Impression, Strength)]` | 技能标签（印象 + 强度） |
| `[IsInfinite]` | 标记为无限持续技能 |

## 6. 技能 DSL

技能不直接修改战场状态，而是先翻译为 `IDSLSourceFile`（`SourceFile`），再通过 `Compile(JudgeRuleManager?)` 生成 `Intent`，最终在判定管线中执行。

### 6.1 核心数据接口：`IAnalyzableData`

```csharp
public interface IAnalyzableData
{
    string AnalyzerKey { get; }
}
```

所有在判定管线中流转的数据都实现此接口。`AnalyzerKey` 字符串指向 `AnalyzerRegistry` 中注册的静态分析方法。

四种标准 AnalysableData：

| 数据类型 | 写入语句 | 关键字段 |
|---|---|---|
| `AttackAnalyzableData` | `WriteAttack` | `AnalyzerKey`, `Type` (AttackType), `Power` (int), `Clock`, `APFactor`, `TotalDamage`, `StageKeys`, `ExtraParams` |
| `DefenseAnalyzableData` | `WriteDefense` | `AnalyzerKey`, `Defense` (DefenseEntity), `Clock` |
| `ResourceAnalyzableData` | `WriteResource` | `AnalyzerKey`, `Type` (ResourceType), `Power` (float), `Clock` |
| `EffectAnalyzableData` | `WriteEffect` | `AnalyzerKey`, `Type` (EffectType), `TargetType`, `Power`, `Clock`, `EntityClock` |

此外 `EffectEntity` 也实现 `IAnalyzableData`（见第 9 节）。

### 6.2 SourceFile 结构

```csharp
public class SourceFile : IDSLSourceFile
{
    // 核心数据
    Dictionary<Type, SourceFile> _states;         // 状态切换：SourceFile ↔ AttackFile/DefenseFile/...
    List<Sentence> _sentences;                    // 有序操作列表
    Stack<Sentence> _rhetoricCache;               // 修辞缓存（WithCallback/WithModify 等推迟操作）
    List<List<ICallbackOnJudge>> _callbacksOnCompile; // 待注册的动态回调
    
    Community _owner;  // （隐式）施放者
    bool IsPassive;    // 是否来自被动技能
}
```

**Sentence 结构**：

```csharp
class Sentence
{
    Action<Community> Structure;   // 执行体（匿名函数）
    SentenceType SentenceType;     // Attack / Defense / Resource / Effect / Recovery / Free
    StructureType StructureType;   // Main（主语句） / Rhetoric（修辞，挂载到前一个 Main 后）
    Sentence? BindSentence;        // 修辞绑定的主语句
}
```

### 6.3 `SwitchState<TFile>()` 状态切换

链式调用中，`WriteXxx` 方法会切换 `SourceFile` 的类型状态，使后续链式方法仅对特定类型可用：

```
SourceFile
  .WriteAttack(...)   → AttackFile   （可用 WithModify / WithCallback / WithBloodSuck / WithInterupt）
  .WriteDefense(...)  → DefenseFile  （可用 WithModify）
  .WriteResource(...) → ResourceFile （可用 WithModify）
  .WriteEffect(...)   → EffectFile   （可用 WithModify）
  .WriteRecovery(...) → RecoveryFile （可用 WithModify）
```

状态切换通过内部 `_states` 字典实现，所有状态共享同一份 `_sentences`、`_rhetoricCache` 和 `_callbacksOnCompile`。

`WithModify` 是所有 File 类型的通用修辞方法，签名为 `Action<AttackAnalyzableData>`——它始终修改同一回合内最近写入的 `AttackAnalyzableData`。

### 6.4 Pen 模式

```csharp
using Pen = Func<BlacksmithDSL.SourceFile, BlacksmithDSL.SourceFile>;
using DSL = BlacksmithDSL;

Pen pen = sf => sf
    .UseResource(1, ResourceType.Instance.Iron())
    .WriteAttack(3, AttackType.Instance.Physical())
    .WithBloodSuck(0.66f);

return DSL.CreateBy(pen);
```

`DSL.CreateBy(pen)` 等价于 `pen(new SourceFile())`——创建空的 SourceFile，应用 Pen 链。

### 6.5 Write 方法完整签名

#### WriteAttack

```csharp
public AttackFile WriteAttack(
    int power,                                              // 攻击力（整数）
    AttackType.CEValue attackType,                          // 攻击类型
    int delayRounds = 0,                                    // 延迟回合
    float APFactor = 1f,                                    // 穿甲因子
    string analyzerKey = nameof(StandardAnalyzers.DefaultAttack),  // 分析器键
    Func<bool>? ifUndo = null                               // 撤销条件
)
```

#### WriteDefense

```csharp
public DefenseFile WriteDefense(
    DefenseEntity defense,                                  // 防御实体
    int delayRounds = 0,
    string analyzerKey = nameof(StandardAnalyzers.DefaultDefense),
    Func<bool>? ifUndo = null
)
```

注意：`WriteDefense` 不再需要单独的 `power` 参数——`DefenseEntity.Power` 已携带防御值。

#### WriteResource

```csharp
public ResourceFile WriteResource(
    float power,                                            // 资源量（浮点）
    ResourceType.CEValue type,
    int delayRounds = 0,
    string analyzerKey = nameof(StandardAnalyzers.DefaultResource),
    Func<bool>? ifUndo = null
)
```

#### WriteEffect

```csharp
public EffectFile WriteEffect(
    EffectType.CEValue type,                 // 效果触发阶段
    EffectTargetType.CEValue targetType,     // 目标类型（己方/敌方）
    ClapRoundClock entityClock,              // 效果实体的生命周期时钟
    string analyzerKey,                      // 分析器键（无默认值）
    int delayRounds = 0,
    float power = 0,
    Func<bool>? ifUndo = null
)
```

#### WriteRecovery

```csharp
public RecoveryFile WriteRecovery(
    int power,
    Func<bool>? ifUndo = null
)
```

直接回复 HP（编译时执行，不经过分析器）。

### 6.6 `ifUndo` 撤销机制

所有 Write 方法和 `UseResource`/`LoseHP`/`LoseMHP` 都接受可选的 `ifUndo` 参数。在 `Intent.Execute` 执行到该 Sentence 时，如果 `ifUndo()` 返回 `true`，则该 Sentence 被跳过。

`ifUndo` 的核心用途是配合 Mark 系统的 `out Lazy<T>` 实现条件性执行（详见 6.9 节）。

### 6.7 AttackFile 链式方法

#### WithModify

```csharp
public AttackFile WithModify(Action<AttackAnalyzableData> modifier)
```

修改同一回合最近写入的 `AttackAnalyzableData`。作为修辞（Rhetoric）挂载到对应的 WriteAttack 语句之后执行。

典型用法——结合 Mark 计数增加攻击力：
```csharp
.TakeMark(nameof(TripleStrike), out var layerNum)
.WriteAttack(11, AttackType.Instance.Physical())
    .WithModify(last => last.Power += layerNum.Value);
```

#### WithCallback

```csharp
public AttackFile WithCallback(AttackStage.CEValue stage, string analyzerKey)
```

在攻击结算的特定阶段触发分析器。`AttackStage` 有三个阶段：

| AttackStage | 触发时机 |
|---|---|
| `OnHitArmorFirstTime` | 首次命中护甲时 |
| `OnHitBody` | 穿透所有防御后、扣血前 |
| `OnEnd` | 攻击结算完毕后 |

#### WithBloodSuck

```csharp
public AttackFile WithBloodSuck(float percent)
```

便捷方法，等价于：
```csharp
.WithModify(last => last.ExtraParams[nameof(StandardAnalyzers.DefaultBloodSuck)] = percent)
.WithCallback(AttackStage.Instance.OnEnd(), nameof(StandardAnalyzers.DefaultBloodSuck))
```

攻击结算后在 `OnEnd` 阶段按 `TotalDamage * percent` 吸血。

#### WithInterupt

```csharp
public AttackFile WithInterupt()
```

便捷方法，等价于：
```csharp
.WithCallback(AttackStage.Instance.OnHitArmorFirstTime(), nameof(StandardAnalyzers.DefaultInterupt))
```

首次命中护甲时移除对手的 Iron/Gold_Iron/Magic 资源数据。

### 6.8 WriteFree 与编译时动作

```csharp
public SourceFile WriteFree(Action<Community> action, Func<bool>? ifUndo = null)
```

向 `_sentences` 中插入一个 `SentenceType.Free` 的 Sentence。`WriteFree` 不触发状态切换（返回 `SourceFile`），可在任意位置调用。

以下便捷方法内部实现为 `WriteFree`：

| 方法 | 等价于 |
|---|---|
| `UseResource(need, type)` | `WriteFree(source => source.Focus.Get<Resource>().Use(type, need))` |
| `LoseHP(loss)` | `WriteFree(source => source.Focus.Get<Health>().LoseHP(loss))` |
| `LoseMHP(loss)` | `WriteFree(source => source.Focus.Get<Health>().LoseMHP(loss))` |
| `AddMark(markName)` | `WriteFree(source => source.Focus.Get<Effect>().Add(entity))` |
| `TakeMark(...)` | `WriteFree(source => ...)` |
| `CountMark(...)` | `WriteFree(source => ...)` |
| `RegistCallbackOnJudge(callbacks)` | `WriteFree(source => ...)` |

### 6.9 Mark 系统

Mark 是存储于 Effect 组件的无限期 `EffectEntity`（`IsMark = true`, `Type = Default`, `isInfinite: true`），按 `AnalyzerKey`（即 markName）标识和查找。它是跨回合状态管理的核心机制。

#### DSL 链式方法（声明时注册，执行时生效）

**AddMark** — 写入一个标记：
```csharp
public SourceFile AddMark(string markName)
```

**TakeMark** — 取出所有同名标记并返回数量：
```csharp
public SourceFile TakeMark(string markName, out Lazy<int> layerNum)
public SourceFile TakeMark(IReadOnlySet<string> markNames, out Lazy<IReadOnlyDictionary<string, int>> layerNums)
```

**CountMark** — 计数但不移除：
```csharp
public SourceFile CountMark(string markName, out Lazy<int> layerNum)
```

#### Body 扩展方法（立即执行，用于 Analyzer/JudgeCallback 内部）

```csharp
body.AddMark(markName)                         // 立即写入
int count = body.CountMark(markName)           // 立即计数
int taken = body.TakeMark(markName)            // 立即取出
```

#### 数据冒险 ★

`out Lazy<T>` 是一个延迟求值包装器。**其值仅在 `Intent.Execute` 执行到达对应 Sentence 时才可用。** 在 Pen 声明阶段或 Sentence 尚未执行时访问 `.Value` 会抛出 `InvalidOperationException`。

**错误用法**（在声明阶段访问）：
```csharp
Pen pen = sf =>
{
    sf.TakeMark("myMark", out var layerNum);
    int count = layerNum.Value;  // ❌ InvalidOperationException!
    return sf.WriteAttack(count, AttackType.Instance.Physical());
};
```

**正确用法**（在延迟执行的 lambda 中访问）：
```csharp
Pen pen = sf => sf
    .TakeMark(nameof(TripleStrike), out var layerNum)
    .WriteAttack(11, AttackType.Instance.Physical())
        .WithModify(last => last.Power += layerNum.Value);  // ✅ 执行到此时 TakeMark 的 Sentence 已完成
```

`WithModify` 作为修辞（Rhetoric）被插入到绑定的 WriteAttack 之后执行，因此在 `WithModify` 的 lambda 中，`TakeMark` 的 Sentence（写在前面的 Main 语句）已经执行完毕，`layerNum.Value` 安全可用。

#### 条件性 ifUndo 配合 Mark

`ifUndo` 与 `out Lazy<T>` 结合可实现"根据 Mark 计数决定是否执行本条语句"：

```csharp
.TakeMark(new HashSet<string> { "FireMark", "IceMark" }, out var layerNums)
.WriteAttack(3, AttackType.Instance.Physical(),
    ifUndo: () => layerNums.Value["FireMark"] <= 0)   // 没有 FireMark 则跳过
.WriteDefense(new CommonArmor { Power = 2 }, 
    ifUndo: () => layerNums.Value["IceMark"] <= 0)    // 没有 IceMark 则跳过
```

注意：`ifUndo` 的 lambda 同样只在 Execution 阶段求值，此时 `layerNums.Value` 已安全可用。

### 6.10 Compile(judgeRuleManager) → Intent

```csharp
public Intent Compile(JudgeRuleManager? judgeRuleManager = null)
```

编译分三步：
1. **展开修辞缓存**：弹出 `_rhetoricCache` 中的修辞 Sentence，按 `BindSentence` 定位插入到对应主语句之后
2. **合并句子**：将所有 `Sentence.Structure` 合并为单个 `Action<Community>`（多播委托）
3. **注册动态回调**：如果 `judgeRuleManager != null`，在合并后的委托末尾追加 `_callbacksOnCompile` 的注册逻辑

返回 `Intent { Execute = result }`——执行它就按序运行所有 Sentence。

### 6.11 RegistCallbackOnJudge

```csharp
public SourceFile RegistCallbackOnJudge(List<ICallbackOnJudge> callbacks)
```

技能 DSL 中注册动态判定规则。回调在 `Compile` 时落地到 `JudgeRuleManager`。详见[判定流程](./判定流程.md#6-动态注册)。

## 7. Analyzer 体系

AnalyzerRegistry + `[IsAnalyzer]` 是 HPB 的核心基础设施，实现数据与行为的完全解耦。

### 7.1 架构概览

```
┌─────────────────────────────────────────────────┐
│                  AnalyzerRegistry                │
├─────────────────────────────────────────────────┤
│  .DSL          Action<Community, Community,     │
│                IAnalyzableData>                  │
│  .Defense      Action<Community, Community,     │
│                DefenseEntity, AttackAnalyzableData>│
│  .JudgeCallback Action<Community, Community>    │
│  .Universal    UniversalRegistry                │
└─────────────────────────────────────────────────┘
         ▲                ▲
         │ 注册            │ 查找
         │                │
 ┌───────┴──────┐  ┌──────┴──────┐
 │ [IsAnalyzer] │  │ AnalyzerKey │
 │ 静态方法      │  │ 字符串       │
 └──────────────┘  └─────────────┘
```

### 7.2 AnalyzerType 枚举

```csharp
public enum AnalyzerType
{
    DSL,           // 通用：Attack/Defense/Resource/Effect 写入与执行
    Defense,       // 防御专属：签名含 DefenseEntity + AttackAnalyzableData
    JudgeCallback, // 判定回调
    Universal      // 泛型注册
}
```

### 7.3 [IsAnalyzer] 属性

```csharp
[AttributeUsage(AttributeTargets.Method, AllowMultiple = false, Inherited = false)]
public class IsAnalyzer : Attribute
{
    public readonly AnalyzerType Type;
    public IsAnalyzer(AnalyzerType type) { Type = type; }
}
```

标记了 `[IsAnalyzer]` 的静态方法在 Mod 加载时由源生成器 `AnalyzersRegistrationGenerator` 自动扫描并注册到对应的 `AnalyzerRegistry` 中。注册键为方法名（`nameof(Class.Method)`）。

另有 `[IsAnalyzerAlias("TargetMethod")]` 用于在主职业技能方法上声明"该方法复用已有分析器的逻辑"。

### 7.4 标准分析器一览

**StandardAnalyzers.DSL.cs**（签名 `Action<Community, Community, IAnalyzableData>`）：

| Analyzer | 功能 |
|---|---|
| `DefaultAttack` | 完整攻击流程：遍历防御 → AP 因子穿透 → 触发阶段回调 → 扣血 |
| `DefaultBloodSuck` | 吸血（由 WithBloodSuck 挂载到 OnEnd 阶段） |
| `DefaultInterupt` | 打断：移除 Iron/Gold_Iron/Magic 资源数据 |
| `DefaultResource` | 资源获取：`Resource.Gain(type, power)` |
| `DefaultDefense` | 防御添加：`Defense.Add(defenseData.Defense)` |

**StantardAnalyzers.Defense.cs**（签名 `Action<Community, Community, DefenseEntity, AttackAnalyzableData>`）：

| Analyzer | 功能 |
|---|---|
| `DefaultArmor` | 护甲吸收伤害，护甲值同步减少 |
| `DefaultReduction` | 减伤，值不减少，纯吸收 |
| `ThornReduction` | 荆棘减伤，吸收物理攻击时反弹 50%（向上取整）为魔法伤害（1 回合延迟） |
| `MagicalImmunity` | 魔法免疫，魔法攻击伤害归零 |
| `PhysicalImmunity` | 物理免疫，物理攻击伤害归零 |
| `PercentageReduction` | 百分比减伤：`attackDamage * defensePower / 100`（向上取整） |

**Universal**：
| Analyzer | 签名 | 功能 |
|---|---|---|
| `DefaultMerge` | `Action<DefenseEntity, DefenseEntity>` | 默认合并策略（空操作） |

### 7.5 关键收益

- **新增类型极简**：只需声明数据类 + 注册 Analyzer
- **行为可动态替换**：更换 `AnalyzerKey` 即可改变行为
- **计算逻辑集中审计**：所有逻辑在 `StandardAnalyzers` 中一目了然
- **减少委托分配**：静态方法引用替代闭包，降低 GC 压力

## 8. 判定引擎

完整 9 阶段执行流程见 [判定流程](./判定流程.md)。

### 完整链路

```
SkillPackage → IDSLSourceFile → Compile(judgeRuleManager) → Intent
→ TurnContext (AnalyzableDatas) → JudgeRuleManager (9-Stage Chain) → Body.GetView()
```

### 9 个判定阶段

| 优先级 | 阶段 | 默认规则 | 说明 |
|---|---|---|---|
| 0 | `OnBegin` | 空 | 留给动态注册——状态检查、Buff 重置 |
| 8 | `OnApplyingEffect` | `ApplyEffect` | 执行 EffectAnalyzableData，创建 EffectEntity 存入目标 Effect 组件 |
| 16 | `OnEffectTaking_AfterAnalyzableDataWritten` | `TakeEffects(AfterAnalyzableDataWritten)` | 执行刚写入的效果 |
| 32 | `OnAttackCanceling` | `CancelAttacks` | 双方攻击按 Type 排序后 1:1 对消，跳过真实伤害和延迟攻击 |
| 64 | `OnEffectTaking_AfterTransport` | `TakeEffects(AfterTransport)` | 执行传输后生效的效果 |
| 128 | `OnApplyingOthers` | `ApplyOthers` | 依次：防御部署 → 攻击结算 → 资源获取。最密集的阶段 |
| 256 | `OnUpdating` | `Update` | `Community.Update()`——清理召唤物、推进时钟、更新组件 |
| 1024 | `OnEffectTaking_AfterResult` | `TakeEffects(AfterResult)` | 结算完成后的效果触发 |
| 2048 | `OnEnd` | 空 | 留给动态注册——回合结束清理重置 |

### 执行方式

所有 AnalysableData 的执行通过 `AnalyzerRegistry` 调度：

```csharp
AnalyzerRegistry.DSL.Get(analyzableData.AnalyzerKey)(player, enemy, analyzableData);
```

防御计算：
```csharp
AnalyzerRegistry.Defense.Get(defense.AnalyzerKey)(enemy, player, defense, attackData);
```

### 动态注册核心类型

| 类型 | 职责 |
|---|---|
| `ICallbackOnJudge` | 接口，定义 `AnalyzerKey`（string）、`Stage`（`JudgeStage.CEValue`）、`Clock`（`ClapRoundClock`）、`IsPlayer`（bool） |
| `ModifierCallback` | 实现 `ICallbackOnJudge`，额外携带 `ModifierOrder`（`Before`/`After`）。作为阶段核心规则的修饰器 |
| `OverrideCallback` | 实现 `ICallbackOnJudge`，直接替换阶段的默认规则 |

### StageRuleContainer 执行顺序

```
Before Modifiers（仅 IsRinging 的）
→ 核心规则（Override 或 Base，通过 AnalyzerRegistry.JudgeCallback 调度）
→ After Modifiers（仅 IsRinging 的）
→ Update()：移除 IsDead 规则，RoundPass() 推进
```

## 9. 防御系统

所有防御继承 `DefenseEntity`：

```csharp
public partial class DefenseEntity : IAnalyzableData
{
    public required string Name { get; init; }         // 防御名称
    public required string AnalyzerKey { get; init; }  // 指向 AnalyzerRegistry.Defense
    public string MergeKey { get; init; }              // 合并策略
    public required DefenseType.CEValue Type { get; init; } // 决定优先级
    public required int Power { get; set; }            // 防御值（整数）
    public required ClapRoundClock Clock { get; init; }
    public bool CanMerge { get; init; } = false;
    public void Update() { Clock.RoundPass(); if (Power <= 0) Clock.Kill(); }
}
```

**防御子类为纯数据声明**——不包含任何 `Work()` 方法。计算逻辑全部在 `StandardAnalyzers` 中。

### 内置防御类型

| 防御类型 | 优先级 | 特性 |
|---|---|---|
| `PhysicalImmunity` | -16 | 物理免疫 |
| `MagicalImmunity` | -16 | 魔法免疫 |
| `RealReduction` | 0 | 真实伤减 |
| `ThornReduction` | 8 | 荆棘伤减，吸收时反弹 50% 为魔法伤害 |
| `CommonReduction` | 16 | 普通伤减 |
| `PercentageReduction` | 可变 | 百分比减伤（`attackPower / 100 * defensePower`） |
| `CommonArmor` | 128 | 普通装甲，耐久型 |

### 攻击结算流程（DefaultAttack 分析器）

1. 真实伤害（`AttackType.Real()`）→ 跳过所有防御直接扣 HP
2. 否则按优先级遍历防御列表：
   - 首次命中 APList 中的护甲类型 → 触发 `OnHitArmorFirstTime` 阶段回调
   - APList 中的防御 → `Power *= APFactor`（穿甲）
   - `AnalyzerRegistry.Defense.Get(defense.AnalyzerKey)(...)` → 防御计算
   - APList 中的防御 → `Power /= APFactor`（还原）
   - `Power <= 0` → 触发 `OnEnd` 回调，结束
3. 触发 `OnHitBody` 回调 → `LoseHP(Power)` → `TotalDamage += Power` → 触发 `OnEnd` 回调

APList：`ThornReduction`, `CommonReduction`, `StoneShell`, `RealArmor`, `CommonArmor`
ArmorList：`StoneShell`, `RealArmor`, `CommonArmor`

## 13. 排障指南

- **技能按钮不可用**：区分 `Illegal`（未注册）和 `Rejected`（Check 不通过）
- **技能未出现**：最常见——返回了 `DSL.SourceFile` 而非 `IDSLSourceFile`（源生成器报告 Error）
- **Mod 未加载**：确认 `mod.json` 配置、DLL 位置、特性标注、`CloseFactory()` 前完成注册
- **结算结果不对**：按链路排查——DSL → Intent → AnalysableData → 阶段规则 → AnalyzerRegistry → 动态注册干扰
- **Analyzer 未生效**：检查 `[IsAnalyzer]` 特性是否标注、AnalyzerType 是否正确、AnalyzerKey 字符串是否匹配
- **`Lazy<T>.Value` 抛出 InvalidOperationException**：Mark 数据在声明时不可用。确保访问 `.Value` 的代码在延迟执行的 lambda（`WithModify`、`ifUndo`、`WriteFree` 等）内部
